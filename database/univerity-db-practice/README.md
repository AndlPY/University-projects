# University DB Practice

This folder contains six SQL practice assignments adapted for PostgreSQL. The exercises are based on a simple academic database context and are meant to train direct command usage: creating tables, inserting data, updating rows, filtering results, joining tables, grouping data, and writing subqueries.

## Learning Focus

The main goal is not business analysis or product-style decision making. These tasks are intentionally procedural: read the instruction, write the SQL command, run it, inspect the result, and adjust the query if needed.

This format is useful for practicing the mechanics of SQL:

- table structure and constraints;
- basic DML operations;
- safe updates and deletes;
- filtering and sorting;
- joins between related tables;
- aggregate functions, grouping, and subqueries.

At the same time, this is a limitation of the assignment style. The tasks do not start from realistic analytical questions such as "find the top 5 students who qualify for an honors diploma" and then derive the SQL from that business goal. They are mostly command drills. That is why the folder should be read as SQL practice material, not as a complete case-study project.

## Why PostgreSQL

The original classroom tasks were close to SQLite-style exercises, but this version is written for PostgreSQL because the implemented portfolio project uses PostgreSQL. The exercises therefore use PostgreSQL-oriented syntax and habits:

- `GENERATED BY DEFAULT AS IDENTITY` for generated keys;
- real `DATE` columns instead of date strings;
- `VARCHAR`, `NUMERIC`, and explicit constraints;
- `RETURNING` for checking changed rows;
- transactions with `BEGIN`, `COMMIT`, and `ROLLBACK`;
- `TRUNCATE TABLE ... RESTART IDENTITY`;
- `ILIKE`, `FILTER`, and `CASE` where they fit naturally.

## How To Work With The Tasks

1. Create a local PostgreSQL database, for example `college_practice`.
2. Open DBeaver, pgAdmin, or `psql`.
3. Go through the practices in order.
4. Keep each solution as a separate SQL script or clearly separated query block.
5. Add short comments before important query groups.
6. Run verification `SELECT` queries after changes.
7. Use transactions when testing risky `UPDATE`, `DELETE`, or `TRUNCATE` statements.
